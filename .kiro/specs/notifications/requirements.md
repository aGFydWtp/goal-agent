# Requirements Document

## Project Description (Input)
中心仮説は「評価目標管理はダッシュボードより、毎週聞いてくれるチャット Agent の方が継続する」である。ユーザーが自発的に `/checkin` するのを待つだけでは継続しない。本機能は、定期的にこちらから週次チェックインを促し、状態が悪化・停滞・期限接近した目標を能動的にアラートすることで、継続利用(成功指標)を支える。

- **誰の課題か**: 半期評価目標を Discord Bot で継続追跡する個人ユーザー。自発的な振り返りを忘れがちで、状態が悪化していても気づけない。
- **現状**: infra-foundation(Agent + `this.schedule()`)、discord-gateway(プロアクティブ送信ヘルパー)、status-and-draft(ステータス判定)が揃っているが、スケジューリングとアラートが未実装。
- **何を変えるか**: 毎週金曜 16:30 に現在の Green/Yellow/Red 件数を含むチェックイン通知を DM(失敗時は個人用非公開チャンネル)へ届ける(§9.1)。状態悪化(Green→Yellow、Yellow→Red)・証跡なし2週間継続・半期終了30日前/14日前のトリガで Red/Yellow アラートを理由と改善導線(`/goal status <id>` の案内)付きで届ける(§9.3)。状態判定は status-and-draft の判定メソッドを消費し、送信は discord-gateway の送信ヘルパーを利用する。状態遷移検出に必要な「直近の判定状態」は本機能が保持する。

## Boundary Context
- **In scope**: 週次チェックイン通知のスケジューリング(`this.schedule()` cron、初期 毎週金曜 16:30)、チェックイン通知文の組み立て(現在の Green/Yellow/Red 件数付き §9.1)、Red/Yellow アラートのトリガ評価(Green→Yellow / Yellow→Red の状態遷移、証跡なし2週間継続、半期終了30日前/14日前 §9.3)、状態遷移検出に必要な「目標ごとの直近判定状態」の保持と比較、アラート文の組み立て(理由 + `/goal status <id>` 改善導線)、DM 失敗時の個人用非公開チャンネルへのフォールバック配信。
- **Out of scope**: ステータス判定ロジック本体(status-and-draft の判定メソッドを呼ぶ)、`/checkin` の会話・分類・証跡化(checkin-classification)、Google Calendar 連携・`/prepare 1on1`(MVP 対象外・将来枠 §9.2)、プロアクティブ送信ヘルパーの実装・DM open / 403 フォールバックの REST 機構(discord-gateway)、Agent クラス骨格・`this.schedule()` の基盤・§11 永続化スキーマ・LLM クライアント(infra-foundation)。
- **Adjacent expectations**: infra-foundation が Agent(EvaluationCycleAgent / GoalAgent)・`this.schedule()` による分粒度 cron・DO SQLite(`this.sql`)・Agent 取得ルーティングを提供すること、discord-gateway が DM open → 送信 → 403 時の個人用非公開チャンネルフォールバック → 判別可能な失敗返却を行う送信ヘルパー(`sendDirectMessage`)を提供すること、status-and-draft が `determineAllStatuses`(全目標集約判定)/ `determineGoalStatus`(単一目標判定)を Agent メソッドとして公開し `StatusVerdict`(状態・理由・risks・nextActions)を返すこと、status-and-draft はステータス判定をオンデマンドで算出し**永続化しない**ため本機能が状態遷移検出のための直近判定状態を保持することを前提とする。

## Requirements

### Requirement 1: 週次チェックイン通知のスケジューリング
**Objective:** 評価目標を追跡するユーザーとして、自分から振り返らなくても毎週決まった時刻にチェックインを促してほしい。そうすれば、継続的に進捗を記録できる。

#### Acceptance Criteria
1. When 評価サイクルを持つユーザーの通知スケジュールが初期化される, the 週次チェックインスケジューラ shall 毎週金曜 16:30 に発火する定期スケジュールを登録する。
2. When 週次チェックインのスケジュールが発火した, the 週次チェックインスケジューラ shall 対象ユーザーのアクティブな評価サイクルについてチェックイン通知の組み立てと配信を起動する。
3. While 週次チェックインのスケジュールが登録されている, the 週次チェックインスケジューラ shall 次回以降も毎週同一曜日・時刻に繰り返し発火するようスケジュールを維持する。
4. If 同一の週次チェックインスケジュールが既に登録された状態で再初期化が要求された, then the 週次チェックインスケジューラ shall 重複した通知が同一週に複数回送られないようスケジュールを冪等に保つ。
5. If 発火時点で対象ユーザーがアクティブな評価サイクルを持たない, then the 週次チェックインスケジューラ shall チェックイン通知を送らずに処理を終了する。

### Requirement 2: 週次チェックイン通知文の組み立てと配信
**Objective:** 評価目標を追跡するユーザーとして、チェックイン通知で今の全体状況(状態件数)も一緒に把握したい。そうすれば、今週どこに注力すべきかを意識して入力できる。

#### Acceptance Criteria
1. When 週次チェックイン通知の組み立てが起動された, the チェックイン通知サービス shall 対象サイクルの全目標について status-and-draft の全目標判定を実行し、Green・Yellow・Red の件数を集計する。
2. When 件数集計が完了した, the チェックイン通知サービス shall §9.1 のチェックイン文(今週やったことの入力を促す文面)に現在の Green/Yellow/Red の件数を含めた通知メッセージを組み立てる。
3. When チェックイン通知メッセージが組み立てられた, the チェックイン通知サービス shall discord-gateway の送信ヘルパーを用いて対象ユーザーへプロアクティブに配信する。
4. If 対象サイクルに目標が1件も登録されていない, then the チェックイン通知サービス shall 件数を全て0件として扱い、チェックイン文を配信する。
5. The チェックイン通知サービス shall チェックイン通知を実行ユーザー本人のみが受け取る経路(DM または本人の個人用非公開チャンネル)に限定して配信する。

### Requirement 3: 直近判定状態の保持と状態遷移検出
**Objective:** アラート機能の提供者として、状態が悪化したことを検出するために前回の状態を覚えておきたい。そうすれば、Green→Yellow・Yellow→Red の遷移を正しく検出できる。

#### Acceptance Criteria
1. The アラート評価エンジン shall 目標ごとの直近の判定状態(Green/Yellow/Red/Gray)を、所有ユーザーと対応づけて保持する。
2. When アラート評価のために最新の状態判定が実行された, the アラート評価エンジン shall 新たに算出した状態を保持中の直近状態と比較して状態遷移を判定する。
3. When 状態遷移の判定が完了した, the アラート評価エンジン shall 比較に用いた新しい状態を当該目標の直近状態として更新・保持する。
4. If ある目標について直近状態がまだ保持されていない(初回判定), then the アラート評価エンジン shall その状態を悪化遷移とはみなさず、当該状態を初期の直近状態として保持する。
5. The アラート評価エンジン shall 直近判定状態の比較・更新を、status-and-draft が永続化しない判定結果に依存せず、本機能が保持する状態を唯一の比較元として行う。

### Requirement 4: Red / Yellow アラートのトリガ評価
**Objective:** 評価目標を追跡するユーザーとして、状態が悪化・停滞・期限接近したときに能動的に知らせてほしい。そうすれば、手遅れになる前に打ち手を取れる。

#### Acceptance Criteria
1. When アラート評価が起動された, the アラート評価エンジン shall 対象ユーザーのアクティブサイクルの各目標について最新の状態判定を取得する。
2. If ある目標の状態が Green から Yellow へ遷移した, then the アラート評価エンジン shall その目標について Yellow アラートのトリガが成立したと判定する。
3. If ある目標の状態が Yellow から Red へ遷移した, then the アラート評価エンジン shall その目標について Red アラートのトリガが成立したと判定する。
4. If ある目標について証跡なしの状態が2週間継続している, then the アラート評価エンジン shall その目標についてアラートのトリガが成立したと判定する。
5. If 対象サイクルの半期終了まで残り30日となった, then the アラート評価エンジン shall 期限接近のアラートのトリガが成立したと判定する。
6. If 対象サイクルの半期終了まで残り14日となった, then the アラート評価エンジン shall 期限接近のアラートのトリガが成立したと判定する。
7. The アラート評価エンジン shall 半期終了までの残り日数を対象サイクルの終了日と現在日付から算出する。
8. If 同一のトリガ条件について既に同一サイクル内でアラートを送信済みである, then the アラート評価エンジン shall 同じトリガで重複してアラートを送らない。

### Requirement 5: アラート通知文の組み立てと配信
**Objective:** 評価目標を追跡するユーザーとして、アラートで何が問題でどう対処すればよいかを知りたい。そうすれば、すぐ次の行動に移れる。

#### Acceptance Criteria
1. When ある目標についてアラートのトリガが成立した, the アラート通知サービス shall 対象目標の名称・新しい状態・成立した理由(状態悪化 / 証跡なし継続 / 半期終了までの残り日数 等)を含むアラートメッセージを §9.3 形式で組み立てる。
2. The アラート通知サービス shall アラートメッセージに改善導線として当該目標を対象とする `/goal status <id>` の案内を含める。
3. When アラートメッセージが組み立てられた, the アラート通知サービス shall discord-gateway の送信ヘルパーを用いて対象ユーザーへプロアクティブに配信する。
4. The アラート通知サービス shall アラート通知を実行ユーザー本人のみが受け取る経路(DM または本人の個人用非公開チャンネル)に限定して配信する。

### Requirement 6: プロアクティブ配信のフォールバックと失敗処理
**Objective:** 評価目標を追跡するユーザーとして、DM が無効でも通知を受け取りたい。そうすれば、設定の都合で通知を取りこぼさない。

#### Acceptance Criteria
1. When 週次チェックインまたはアラートの配信が起動された, the 通知配信サービス shall discord-gateway の送信ヘルパーへ対象ユーザー ID・メッセージ・個人用フォールバックチャンネルを渡して配信を試みる。
2. If DM への配信が拒否(403)され、かつ個人用フォールバックチャンネルが指定されている, then the 通知配信サービス shall フォールバックチャンネルへの配信に切り替える(送信ヘルパーの挙動を利用する)。
3. If DM への配信が拒否され、かつフォールバックチャンネルが指定されていない, then the 通知配信サービス shall 配信失敗を判別可能に記録し、処理を中断せずに終了する。
4. If 配信が失敗した, then the 通知配信サービス shall 該当トリガのアラートを「送信済み」として記録せず、次回の評価機会で再送できる状態を保つ。

### Requirement 7: 境界の維持
**Objective:** 本機能の実装者として、上流が確立した契約を再定義せず消費したい。そうすれば、責務の重複や契約のずれを防げる。

#### Acceptance Criteria
1. The 通知機能 shall ステータス判定を自前で実装せず、status-and-draft が公開する全目標判定/単一目標判定メソッドを呼び出して状態を取得する。
2. The 通知機能 shall プロアクティブ送信の REST 機構(DM open・403 フォールバック)を自前で実装せず、discord-gateway の送信ヘルパーを利用する。
3. The 通知機能 shall 定期実行の基盤(`this.schedule()` / Agent / DO SQLite)・§11 永続化スキーマ・LLM クライアントを infra-foundation の提供物として利用し、再定義しない。
4. The 通知機能 shall `/checkin` の会話処理・分類・証跡化(checkin-classification)および Google Calendar 連携・`/prepare 1on1`(将来枠)を実装しない。
